# Workflows Visual Editor — Design

**Date:** 2026-04-17
**Status:** Design approved, ready for implementation planning

## 1. Overview

Replace the placeholder on `/admin/workflows` with a real visual editor for building and operating workflow definitions. The editor is a canvas-based graph builder with auto-layout that emits a `graph_definition` JSON the existing `WorkflowEngineService` already knows how to execute. It also adds three operational features on top of editing: a runtime viewer (see where a running instance is), an execution history timeline, and a dry-run simulator.

## 2. Goals

- Admins can visually author a workflow without editing JSON.
- All 10 node types the backend engine supports are first-class in the editor.
- Drafts, publishing, and unpublishing work without data loss.
- Published workflows are observable: users can see where any running instance currently is and what path it took.
- Admins can test-run a workflow against a synthetic ticket before publishing.

## 3. Non-goals

- Multi-user collaborative editing (presence, CRDT, locking). Deferred.
- Workflow templates / marketplace / sharing across tenants.
- Arbitrary scriptable expressions or a free-form formula language.
- Versioning with simultaneous running v1 and v2 instances — "unpublish to edit" is the chosen lifecycle.
- Trigger-binding UI (deciding when a workflow fires based on ticket events). Today `startForTicket` is called externally; this spec does not change that.

## 4. Scope summary

| Area | In scope |
|---|---|
| Node types | `trigger`, `end`, `assign`, `approval`, `notification`, `condition`, `update_ticket`, `create_child_tasks`, `wait_for`, `timer` |
| Editor UX | Canvas + dagre auto-layout, palette, inspector, toolbar |
| Interactions | Add, delete, connect, reconnect, copy/paste/duplicate, undo/redo, keyboard shortcuts |
| Validation | Client + server, with surfaced errors |
| Lifecycle | Save draft · Publish · Unpublish · Clone |
| Runtime | Read-only viewer overlaying `workflow_instances.current_node_id` + past-node trail |
| History | `workflow_instance_events` table · engine instrumentation · timeline UI |
| Simulation | Dry-run engine mode · test panel in editor |

## 6. Data model

### 6.1 `graph_definition` JSON shape (existing, formalized here)

Persisted exactly as the engine consumes today; extra fields are ignored by the engine so client-only data (positions, labels) is safe to include.

```jsonc
{
  "nodes": [
    {
      "id": "n_start",           // stable string id
      "type": "trigger",         // one of the 10 supported types
      "config": { /* type-specific */ },
      "position": { "x": 0, "y": 0 }  // client-only, regenerated by dagre if absent
    }
  ],
  "edges": [
    {
      "from": "n_start",
      "to": "n_assign",
      "condition": "true"        // optional; used for `condition` branches ("true"/"false") and `approval` outcomes ("approved"/"rejected")
    }
  ]
}
```

### 6.2 Per-node config schemas

```ts
trigger:            { label?: string }                                   // no runtime config; marks start
end:                { label?: string }
assign:             { team_id?: uuid, user_id?: uuid }                   // at least one required
approval:           { approver_person_id?: uuid, approver_team_id?: uuid }  // at least one required
notification:       { notification_type?: string, subject: string, body: string }  // engine hardcodes in-app channel today
condition:          { field_key: string, operator: string, value: unknown }
update_ticket:      { fields: Record<string, unknown> }                  // e.g. { status: 'in_progress', priority: 'high' }
create_child_tasks: { tasks: Array<{ title, description?, assigned_team_id?, interaction_mode?, priority? }> }
wait_for:           { wait_type: 'child_tasks'|'status'|'event' }
timer:              { delay_minutes: number }
```

### 6.2a Condition field registry (amendment 2026-04-23)

The current implementation is too weak if it only compares a small hardcoded list of raw ticket columns with a free-text input.

The editor must use a **condition field registry**. Each field entry defines:

- `key`
- `label`
- `source`
- `data_type`
- supported operators
- lookup source when values come from real entities
- optional normalization strategy

Example shape:

```ts
type WorkflowConditionField = {
  key: string;
  label: string;
  source: 'ticket' | 'request_type' | 'requester' | 'requested_for' | 'asset' | 'location' | 'assignment';
  data_type: 'enum' | 'text' | 'number' | 'boolean' | 'uuid_ref' | 'uuid_array';
  operators: string[];
  lookup?: {
    entity: 'spaces' | 'teams' | 'vendors' | 'request_types' | 'asset_types';
    endpoint: string;
    label_field: string;
    value_field: string;
  };
  normalize?: 'location_scope' | 'asset_type' | 'request_type' | 'none';
};
```

Minimum useful field set:

- Ticket:
  - `priority`
  - `status`
  - `status_category`
  - `interaction_mode`
  - `source_channel`
- Request:
  - `request_type_id`
  - `domain`
- Location:
  - `location_id`
  - `site_id`
  - `building_id`
- Asset:
  - `asset_id`
  - `asset_type_id`
  - `has_asset`
- People:
  - `requester_person_id`
  - `requested_for_person_id`
  - requester/requested-for attributes such as `department`, `division`, `type`
- Assignment:
  - `assigned_team_id`
  - `assigned_vendor_id`

This is still a bounded condition system. It is not an open scripting engine.

### 6.2b Typed operators

Operators must be driven by field type, not shown as one global flat list.

- Enum / reference:
  - `equals`
  - `not_equals`
  - `in`
  - `not_in`
- Boolean:
  - `is_true`
  - `is_false`
- Text:
  - `equals`
  - `not_equals`
  - `contains`
  - `not_contains`
- Number:
  - `equals`
  - `not_equals`
  - `gt`
  - `gte`
  - `lt`
  - `lte`
- Location/reference hierarchy:
  - `equals`
  - `not_equals`
  - `within_scope_of`
  - `not_within_scope_of`
- Nullable/reference:
  - `exists`
  - `not_exists`

### 6.2c Value picker UX

The editor UX should be metadata-driven:

1. User selects a condition field.
2. The UI loads the allowed operators for that field type.
3. The value control changes automatically:
   - enum -> select
   - boolean -> toggle
   - reference -> async combobox loading real rows
   - multi-reference -> multi-select
   - text/number -> typed input

For reference-backed fields, values should come from real data sources instead of free-text typing:

- location -> spaces lookup
- team -> teams lookup
- vendor -> vendors lookup
- request type -> request types lookup
- asset type -> asset types lookup

This is materially better UX and avoids invalid ids, naming drift, and typo-driven conditions.

### 6.3 Edge semantics (existing, formalized)

- Default flow: first outgoing edge with no `condition` (or the first edge when nothing matches).
- `condition` node emits `"true"` or `"false"`; the editor requires both edges for validation.
- `approval` node emits `"approved"` or `"rejected"` on resume; the editor requires both edges for validation.
- All other node types have exactly one outgoing edge, unconditional.

## 7. Backend changes

### 7.1 `WorkflowValidatorService` (new)

Pure functions; no DB access. Exposed for both publish-time enforcement and pre-save UI feedback.

```ts
validate(graph: WorkflowGraph): { ok: boolean; errors: ValidationError[] }
```

**Validation rules (all enforced):**

| Rule | Severity |
|---|---|
| Exactly one `trigger` node | error |
| At least one `end` node | error |
| Every edge's `from` and `to` reference an existing node | error |
| Every node except `end` has ≥1 outgoing edge | error |
| Every node except `trigger` is reachable from `trigger` (BFS) | error |
| Every path from `trigger` eventually reaches an `end` node (no cycles without exit) | error |
| `condition` nodes have edges labeled both `true` and `false` | error |
| `approval` nodes have edges labeled both `approved` and `rejected` | error |
| `assign` has `team_id` or `user_id` | error |
| `approval` has `approver_person_id` or `approver_team_id` | error |
| `notification` has `subject` and `body` | error |
| `condition` has `field_key`, `operator`, `value` when the chosen operator requires one | error |
| `timer` has `delay_minutes > 0` | error |
| `create_child_tasks` has at least one `tasks[]` entry with a `title` | error |
| `update_ticket` has a non-empty `fields` object | error |
| `wait_for` has `wait_type` set | error |

### 7.1a `WorkflowConditionEvaluatorService` (amendment 2026-04-23)

Condition evaluation should not read only `tickets.*` and compare one raw field.

Introduce a small runtime service that builds a **normalized workflow condition context** before evaluating a condition node.

Example normalized context:

```ts
type WorkflowConditionContext = {
  ticket: {
    id: string;
    priority: string | null;
    status: string | null;
    status_category: string | null;
    interaction_mode: string | null;
    source_channel: string | null;
  };
  request: {
    request_type_id: string | null;
    domain: string | null;
  };
  location: {
    location_id: string | null;
    site_id: string | null;
    building_id: string | null;
    scope_chain: string[];
  };
  asset: {
    asset_id: string | null;
    asset_type_id: string | null;
    has_asset: boolean;
  };
  requester: {
    person_id: string | null;
    type?: string | null;
    department?: string | null;
    division?: string | null;
  };
  requested_for: {
    person_id: string | null;
    type?: string | null;
    department?: string | null;
    division?: string | null;
  };
  assignment: {
    assigned_team_id: string | null;
    assigned_vendor_id: string | null;
  };
};
```

Rules:

- `location_id` should resolve from the effective ticket location, not just `tickets.location_id`. If the ticket has no explicit location but does have an asset with `assigned_space_id`, the context should use that resolved location.
- `site_id` and `building_id` should be derived from the space hierarchy.
- `asset_type_id` should be joined from the linked asset.
- requester/requested-for attributes should come from `persons`.
- assignment values should reflect the current ticket state.

This keeps the builder and the engine aligned: if the UI offers `site_id` or `requester.department`, the engine must know how to evaluate it.

### 7.2 `WorkflowService` additions

```ts
async unpublish(id: string)    // POST /workflows/:id/unpublish
async clone(id: string, name?: string)  // POST /workflows/:id/clone
async listInstanceEvents(instanceId: string)  // GET /workflows/instances/:iid/events
```

- `publish()` calls `validator.validate()` first and throws `BadRequestException` with the error list if invalid.
- `unpublish()` flips `status='draft'` and clears `published_at`. Running instances are unaffected (they re-read the graph on each advance).
- `clone()` inserts a new row with `status='draft'`, `version=1`, `name='<original> (copy)'` (or the provided name), and deep-copies `graph_definition` with regenerated node ids to avoid collision on potential cross-referencing in the future.

### 7.3 `WorkflowEngineService` changes

**Add event emission** at these points (through a small helper that inserts a row):

- `startForTicket` → `instance_started`
- `executeNode` entry → `node_entered` (with node_id, node_type)
- `executeNode` exit (before recursing to `advance`) → `node_exited`
- `condition` branch chosen → `decision_made` with `decision: 'true'|'false'`
- `approval` resume → `decision_made` with `decision: 'approved'|'rejected'`
- Paused states → `instance_waiting` with `payload: { waiting_for }`
- `resume()` → `instance_resumed`
- `end` node → `instance_completed`
- Caught exceptions → `instance_failed` with error in payload

**Add `dryRun` mode:**

Introduce a `WorkflowRunContext` carried through `executeNode` / `advance`:

```ts
interface WorkflowRunContext {
  dryRun: boolean;
  simulatedTicket?: Record<string, unknown>;   // stands in for DB ticket in dry-run
  events: WorkflowEvent[];                     // in-memory collector when dryRun
  stopOnWaiting: boolean;                      // dry-run halts at approval/wait_for/timer
}
```

When `dryRun` is true:
- Skip every `.insert`, `.update`, `.delete` call on Supabase. Replace with event entries in `ctx.events`.
- Substitute ticket reads in `condition` with `ctx.simulatedTicket`.
- On approval/wait_for/timer: push an `instance_waiting` event to `ctx.events` and return (do not update DB).

### 7.4 `WorkflowSimulatorService` (new)

```ts
simulate(workflowId: string, simulatedTicket: Record<string, unknown>): Promise<{
  path: string[];             // node ids in execution order
  events: WorkflowEvent[];
  terminated: boolean;        // reached an `end` node
  stoppedAt?: { node_id: string; node_type: string; reason: 'approval'|'wait_for'|'timer' };
  errors?: string[];
}>
```

Implemented by calling the engine in dry-run mode against the simulated ticket. Returns a deterministic snapshot for the editor's Test panel to render.

### 8.4 State store (`graph-store.ts`)

```ts
interface GraphStore {
  nodes: Node[];
  edges: Edge[];
  selectedId: string | null;
  dirty: boolean;
  past: Snapshot[];
  future: Snapshot[];
  clipboard: { nodes: Node[]; edges: Edge[] } | null;

  setGraph(g: WorkflowGraph): void;
  addNode(type: NodeType, nearId?: string): void;
  updateNodeConfig(id: string, config: object): void;
  deleteSelection(): void;
  connect(from: string, to: string, condition?: string): void;
  disconnect(edgeId: string): void;

  copySelection(): void;
  paste(): void;
  duplicateSelection(): void;

  undo(): void;
  redo(): void;

  reset(): void;
  toJSON(): WorkflowGraph;
}
```

Undo/redo is snapshot-based (cheap for small graphs; ≤100 nodes expected). Every mutating action pushes to `past` and clears `future`; `undo` pops `past` into `future`. Capped at 50 snapshots.

### 8.5 Auto-layout

`layout.ts` wraps dagre:

```ts
applyLayout(nodes, edges): { nodes: Node[] with positions }
```

Called on: initial load, after add/delete/connect, after paste. Direction `LR` (left-to-right). Node size read from measured DOM or sensible defaults (width 200, height 80).

Because layout runs every structural change, positions are re-derived rather than persisted by the user. The `position` field in `graph_definition` is written on save for snapshot fidelity but recomputed on load (always fresh). This avoids drift and stale positions.

## 10. Lifecycle

```
  Draft ── publish (validated) ──▶ Published ── unpublish ──▶ Draft
   │                                   │
   └── clone ──▶ new Draft             └── clone ──▶ new Draft
```

- Edit form is enabled only in `Draft`. When `Published`, editor loads in read-only mode with an "Unpublish to edit" button.
- Running instances are unaffected by unpublish. If the user edits and republishes, in-flight instances read the updated graph on their next `advance` — documented caveat, surfaced in the unpublish confirmation dialog.

## 11. Runtime viewer

Route `/admin/workflows/instances/:instanceId`. Read-only canvas:

- Fetch `workflow_instances` row + its `workflow_definitions` + `workflow_instance_events`.
- Render graph identically to editor, but:
  - Visited nodes (from `node_entered` events) get a subtle filled background.
  - Current node (`workflow_instances.current_node_id`) has a pulsing border and a status label ("Waiting: approval", "Running", "Completed").
  - Edges traversed highlight with a darker stroke.
- Side panel: the `history-timeline` component showing the event log.

Embedded variant: on `ticket-detail.tsx`, if the ticket has workflow instances, render a "Workflow" tab with the compact runtime viewer + timeline.

## 12. Execution history

- Engine instrumented per section 7.3.
- `GET /workflows/instances/:iid/events` returns events ordered by `created_at`.
- `history-timeline.tsx` renders event rows with icons per event type, humanized node labels, and relative timestamps. Used in both runtime viewer and ticket detail.
- No retention policy in v1; audit if table growth becomes a concern.

## 13. Simulate / test-run

- Toolbar "Simulate" opens a sheet/modal with a form:
  - "Simulated ticket" — a small JSON-ish form with the fields `condition` nodes in this graph actually reference (derived from `graph.nodes.filter(n => n.type === 'condition').map(n => n.config.field)`). Users fill values; avoids asking them to construct a full ticket.
  - "Run" button → POST `/workflows/:id/simulate`.
- Result panel:
  - Highlights the traversed path on the canvas (reuses runtime-viewer styling).
  - Renders the event list below.
  - If the run stopped at an approval/wait_for/timer, shows that clearly with a "What happens after?" note.
- No side effects — server enforces `dryRun` mode; no rows written anywhere including `workflow_instance_events` (events are returned in response only, not persisted).

## 14. Clone

- List row action "Clone" → POST `/workflows/:id/clone`, then navigate to the new draft's editor.
- Copy deep-copies `graph_definition` and regenerates node ids to avoid any future cross-reference collisions.
- New name defaults to `"<original> (copy)"` but is editable pre-create via a small prompt.

## 15. Testing approach

- **Backend (Jest via Nest):**
  - `WorkflowValidatorService` — table-driven tests for every rule (valid/invalid graphs).
  - `WorkflowEngineService` dry-run — feed synthetic graphs + tickets, assert `path` and `events`.
  - `WorkflowSimulatorService` — integration-test with real graphs from seed migration.
  - `WorkflowService.clone` / `unpublish` — ensure RLS + tenant isolation.
- **Frontend (Vitest + RTL):**
  - `validation.ts` — same table-driven cases as server.
  - `graph-store.ts` — undo/redo, copy/paste, connect/disconnect cover.
  - Inspector forms — each required-field validation renders.
  - Canvas — basic render + add/delete via store, snapshot a small graph.
- **Manual smoke:**
  - Build a 6-node workflow, publish, start for a real ticket, watch runtime viewer update as approvals are resolved, inspect the timeline.

## 16. Phased delivery (guidance for the planner)

Because the scope spans editor + engine + DB, the implementation plan should be staged so each phase lands a usable increment:

- **Phase A — Editor core + validator + lifecycle:** Canvas, palette, inspector, all 10 node-type forms, validation, save/publish/unpublish/clone. No engine changes.
- **Phase B — Execution history:** `workflow_instance_events` migration, engine instrumentation, timeline component, instance viewer, ticket-detail tab.
- **Phase C — Simulator:** Engine `dryRun` mode, `WorkflowSimulatorService`, simulate panel in editor.

Each phase is independently mergeable and valuable.

## 17. Risks and open questions

- **Dagre re-layout jumpiness** — adding a node can shift sibling positions. Mitigate with animated transitions in React Flow (`fitView` + transition prop). If still jarring, revisit with an optional "Preserve positions" toggle in a later iteration.
- **Event table growth** — every advance emits events. Per-tenant ticket volume is low (IT/facilities domain), so acceptable in v1. Track size; add retention if it exceeds ~1M rows per tenant.
- **Dry-run fidelity** — simulator uses `simulatedTicket` for `condition` evaluation, but other nodes (`update_ticket`) don't branch so their "effect" is just a skipped write — fine. `create_child_tasks` in dry-run emits a synthetic event describing what would be created; not a real ticket. Good enough for V1 "did my path branch correctly" goal.
- **Unpublish side effect on in-flight instances** — documented in the unpublish dialog. If stronger guarantees are needed later, introduce real versioning (section 3 non-goal) as a follow-up spec.
- **Approval `decision` edges** — the engine today uses `edge.condition` to pick the next branch after `resume(edgeCondition)`. The caller of `resume` (approvals UI, wherever it lives) must pass the right string ('approved'/'rejected'). This spec assumes that wiring exists or is added when approvals UI is built; flagging so it's not missed.
