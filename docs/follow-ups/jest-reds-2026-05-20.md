## Per-failure classification

### `visitors/admin.controller.spec.ts`

All three failures live in the `listAll (visibility-bypass)` describe
block. Root cause for all three is commit `cb610e55`
("feat(rls-security-slice-11.5): re-gate visitors/admin — AdminGuard
fully eliminated as a caller"), which moved the `visitors.read_all`
permission gate from an in-body `permissions.requirePermission(req,
'visitors.read_all')` call to the declarative
`@RequirePermission('visitors.read_all')` decorator (enforced by
`PermissionMetadataGuard`). As a side effect `listAll` no longer takes
`@Req() req` as its first parameter — the new signature is `(status?,
buildingId?, limit?)`. The tests still called the method with `req`
as the first positional argument, which shoved the real `status`
value into a discarded fourth slot.

| ID | Test | Decision | Root cause | Post-state |
|---|---|---|---|---|
| V1 | `rejects without visitors.read_all (403)` | B | Permission gate is now decorator-enforced by `PermissionMetadataGuard`; unit-level `new VisitorsAdminController(...)` cannot exercise the guard. | Replaced with `decorator metadata declares visitors.read_all` — asserts `Reflect.getMetadata('requiredPermission', listAll) === 'visitors.read_all'`. Live 403 surface is covered by `pnpm smoke:cross-tenant`. |
| V2 | `happy path: filters on tenant` | B | Test called `listAll(makeReq(), 'expected', BUILDING_ID, '50')` — passing `req` into the `status` slot. Also asserted the now-removed in-body `permissions.requirePermission` call. | Updated to call `listAll('expected', BUILDING_ID, '50')` and assert tenant id is the first SQL parameter. |
| V3 | `clamps limit` | B | Test called `listAll(makeReq(), undefined, undefined, '99999')` — `'99999'` landed in a discarded fourth slot, so `limit` was undefined → default 100, not clamped to 500. | Updated to call `listAll(undefined, undefined, '99999')` — clamp fires, limit asserts 500. |

### `reservations/reservation-edit-scope.spec.ts`

All four failures share a root cause: commit `cf0531d1`
("fix(booking-audit slice4): atomic idempotent split_recurrence_series
(P1-2)") restructured `RecurrenceService.splitSeries`:

- Old signature: `splitSeries(bookingId)`.
- New signature: `splitSeries(bookingId, actor, clientRequestId)`.
- Old behaviour: TS-layer pre-check read `command_operations` BEFORE
  splitSeries to detect retries and skip the side effect (the
  `skipSplitSeries` flag).
- New behaviour: pre-check is GONE — splitSeries is now atomic +
  idempotent in the underlying `split_recurrence_series` RPC (migration
  00411) keyed on `(bookingId, clientRequestId)`. The wrapper is called
  on every commit attempt; on retry the RPC cache-hits and returns the
  SAME `new_series_id` (no orphan series). The RPC owns idempotency
  end-to-end; the TS layer no longer pre-detects retries.

| ID | Test | Decision | Root cause | Post-state |
|---|---|---|---|---|
| E1 | `scope='this_and_following' commit: splitSeries + assembles + RPC + cascade + new_series_id envelope` | B | Asserted `splitSeries.toHaveBeenCalledWith(PIVOT_BOOKING_ID)` — stale 1-arg signature. | Updated assertion to the new 3-arg signature `(bookingId, actor, crid)` with `expect.objectContaining` on the actor + an exact `clientRequestId` match. Added a `Why this changed` comment citing `cf0531d1` + `recurrence.service.ts:1052`. |
| E2 | `C1 (codex-revised): retry on this_and_following commit skips splitSeries; assembler+RPC still run; RPC returns cached envelope` | B | Asserted `splitSeries.not.toHaveBeenCalled()` and 1 `command_operations` pre-check read. Both behaviours retired in slice 4. | Recast as `C1 (slice4 cutover): retry re-invokes splitSeries; the split RPC cache-hits…`. New assertions: splitSeries IS called (with full signature), command_operations reads = 0 (TS pre-check removed). |
| E3 | `C2 (codex-revised): payload_mismatch on retry with different body propagates from RPC` | B | Asserted `splitSeries.not.toHaveBeenCalled()` on payload-mismatch retry. Same retired contract. | Updated to assert `splitSeries` IS called once (the RPC + the payload_mismatch raise from `edit_booking_scope` are unchanged; only the TS-layer skip is gone). Test still verifies that mapped 409 `command_operations.payload_mismatch` surfaces. |
| E4 | `C1 (codex-revised): pre-check read error surfaces command_operations.unexpected_state` | B | The entire code path that emitted `command_operations.unexpected_state` from a pre-check read error no longer exists. The TS layer no longer reads `command_operations`. Test asserts retired behaviour with no replacement surface. | Removed. Replaced with an in-place comment citing `cf0531d1` + the new architecture's locus (`reservation.service.ts:1828-1854` + `recurrence.service.ts:1052`). RPC-level transient errors continue to be covered by `mapRpcErrorToAppError` + the `payload_mismatch` test E3. |

## CI wire-up

Added a `Jest (apps/api)` step to the existing `check` job in
`.github/workflows/ci.yml`, immediately after `Typecheck api`:

```yaml
- name: Jest (apps/api)
  run: pnpm --filter @prequest/api run test
```

The step runs the full apps/api jest suite (191 suites / 2306 tests)
because that suite is currently fully green and the cost is ~25s of
wall time. If a future supabase-mocked spec genuinely needs a live DB,
it should route through the `migration-smoke` or `concurrency-harness`
job, NOT this step — this step is for fast unit-shaped specs.
