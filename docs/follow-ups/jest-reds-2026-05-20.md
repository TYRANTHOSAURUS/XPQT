## Per-failure classification

### `visitors/admin.controller.spec.ts`

All three failures live in the `listAll (visibility-bypass)` describe
block. Root cause for all three is commit `cb610e55`
("feat(rls-security-slice-11.5): re-gate visitors/admin ŌĆö AdminGuard
fully eliminated as a caller"), which moved the `visitors.read_all`
permission gate from an in-body `permissions.requirePermission(req,
'visitors.read_all')` call to the declarative
`@RequirePermission('visitors.read_all')` decorator (enforced by
`PermissionMetadataGuard`). As a side effect `listAll` no longer takes
`@Req() req` as its first parameter ŌĆö the new signature is `(status?,
buildingId?, limit?)`. The tests still called the method with `req`
as the first positional argument, which shoved the real `status`
value into a discarded fourth slot.

| ID | Test | Decision | Root cause | Post-state |
|---|---|---|---|---|
| V1 | `rejects without visitors.read_all (403)` | B | Permission gate is now decorator-enforced by `PermissionMetadataGuard`; unit-level `new VisitorsAdminController(...)` cannot exercise the guard. | Replaced with `decorator metadata declares visitors.read_all` ŌĆö asserts `Reflect.getMetadata('requiredPermission', listAll) === 'visitors.read_all'`. Live 403 surface is covered by `pnpm smoke:cross-tenant`. |
| V2 | `happy path: filters on tenant` | B | Test called `listAll(makeReq(), 'expected', BUILDING_ID, '50')` ŌĆö passing `req` into the `status` slot. Also asserted the now-removed in-body `permissions.requirePermission` call. | Updated to call `listAll('expected', BUILDING_ID, '50')` and assert tenant id is the first SQL parameter. |
| V3 | `clamps limit` | B | Test called `listAll(makeReq(), undefined, undefined, '99999')` ŌĆö `'99999'` landed in a discarded fourth slot, so `limit` was undefined ŌåÆ default 100, not clamped to 500. | Updated to call `listAll(undefined, undefined, '99999')` ŌĆö clamp fires, limit asserts 500. |

### `reservations/reservation-edit-scope.spec.ts`

All four failures share a root cause: commit `cf0531d1`
("fix(booking-audit slice4): atomic idempotent split_recurrence_series
(P1-2)") restructured `RecurrenceService.splitSeries`:

- Old signature: `splitSeries(bookingId)`.
- New signature: `splitSeries(bookingId, actor, clientRequestId)`.
- Old behaviour: TS-layer pre-check read `command_operations` BEFORE
  splitSeries to detect retries and skip the side effect (the
  `skipSplitSeries` flag).
- New behaviour: pre-check is GONE ŌĆö splitSeries is now atomic +
  idempotent in the underlying `split_recurrence_series` RPC (migration
  00411) keyed on `(bookingId, clientRequestId)`. The wrapper is called
  on every commit attempt; on retry the RPC cache-hits and returns the
  SAME `new_series_id` (no orphan series). The RPC owns idempotency
  end-to-end; the TS layer no longer pre-detects retries.

## CI wire-up

Added a `Jest (apps/api) ŌĆö motivating files only` step to the existing
`check` job in `.github/workflows/ci.yml`, immediately after
`Typecheck api`:

```yaml
- name: Jest (apps/api) ŌĆö motivating files only
  run: pnpm --filter @prequest/api run test -- visitors/admin.controller.spec.ts reservations/reservation-edit-scope.spec.ts
```

**Narrow scope, on purpose.** Initial design ran the full
`pnpm --filter @prequest/api run test` because locally the 191-suite
pass is green in ~25s. The first PR-CI run revealed a hidden coupling:
`apps/api/src/modules/visitors/migrations.spec.ts` opens a `pg`
client to validate the RLS predicates on the visitors v1 migration
block (00248ŌĆō00258). Locally the spec's `beforeAll` connects to
Supabase on port 54322 and the suite runs green; in the `check` job
there's no Postgres, so `beforeAll` flips `dbAvailable=false` and the
test bodies skip themselves ŌĆö but its `afterAll` cleanup blindly calls
`client.query("set local role postgres")` regardless, throwing
`Connection terminated unexpectedly`. The whole suite reports failure
even though no F3-touched test fails.

Two paths considered:

1. **Fix `migrations.spec.ts`'s `afterAll` to honour `dbAvailable`.**
   The correct root-cause fix, but out of F3 scope ŌĆö the spec is
   unrelated to either motivating red. Tracked as the follow-up below.
2. **Narrow the CI step to the two motivating files.** Matches the
   F3 brief verbatim (`pnpm -C apps/api test -- visitors/admin.controller.spec.ts reservations/reservation-edit-scope.spec.ts`)
   and closes the silent-drift loop on the two files that were red
   across multiple sessions, without introducing a broader noise
   floor that masks DB-coupled failures in the wrong job.

Path 2 chosen. Once `migrations.spec.ts` (and any sibling DB-coupled
spec that skips `beforeAll` but not `afterAll`) is hardened, broaden
this step ŌĆö don't do it speculatively.
